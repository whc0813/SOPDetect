2 开发技术与工具平台

本章主要介绍系统开发过程中涉及的基础概念、开发工具、技术框架和模型接口。标准操作流程智能评估系统涉及前端页面开发、后端接口服务、视频处理、数据库管理以及多模态模型调用等内容，因此需要了解相关技术的基本概念、主要特点和适用场景。

本章内容按照业务基础、智能分析技术、前端开发工具、后端开发工具、视频处理技术、数据库技术以及大模型接口技术展开，为后续系统分析、系统设计和系统实现部分提供技术基础。

2.1 标准操作流程管理与视频评估概述

2.1.1 SOP 的基本概念

标准操作流程（Standard Operating Procedure，SOP）是对某类工作任务进行规范化描述的操作文件，通常包括任务名称、适用场景、操作步骤、执行顺序、注意事项和质量要求等内容。SOP 的作用在于将经验性操作转化为清晰、稳定、可复制的流程规范，使不同执行者能够按照统一标准完成任务。

在传统管理方式中，SOP 多以文档、表格或培训材料的形式存在。执行人员根据文档要求完成操作，管理人员再通过现场观察、抽查记录或查看视频等方式判断执行结果。随着信息系统的发展，SOP 管理逐渐从静态文档管理转向流程化、数据化和可追踪的管理方式。系统不仅需要保存 SOP 的文字内容，还应记录执行者、执行时间、执行结果、问题证据和复核结论等信息。

在信息化管理场景中，SOP 可以被进一步结构化为流程名称、适用场景、步骤说明、步骤顺序、步骤类型和检查标准等数据内容。结构化后的 SOP 更便于系统存储、查询、展示和评估，也为后续的视频分析和执行结果复核提供了清晰依据。

2.1.2 SOP 执行过程评估需求

SOP 执行过程评估的核心目标是判断执行者是否按照规定步骤完成操作。与普通任务打卡不同，SOP 评估不仅关注任务是否完成，还需要关注完成过程是否符合规范。例如，某一步骤是否漏做，是否只完成了一部分，是否出现动作错误，是否存在顺序颠倒、过早执行、延后执行、重复操作或前置条件缺失等问题。

在实际应用中，SOP 执行过程评估通常具有以下需求。第一，需要支持完整操作过程记录。执行者的一次操作往往是连续过程，完整记录有利于保存真实执行状态。第二，需要支持步骤级结果展示。综合得分只能反映整体表现，不能说明具体错误发生在哪一步，因此评估结果应能够体现步骤完成情况、问题类型和判断依据。第三，需要支持顺序与前置依赖判断。SOP 强调流程顺序，如果用户先做后置步骤再补做前置步骤，即使每个动作单独存在，也不应直接判定为完全合格。第四，需要支持人工复核。智能评估可以降低人工初筛成本，但在关键场景中仍需要保留人工确认和纠偏机制。

因此，SOP 执行过程评估并不是简单的“完成或未完成”判断，而是对操作动作、执行顺序、完成程度、过程证据和最终结论的综合分析。

2.1.3 视频评估在流程规范化管理中的作用

视频能够较完整地记录执行者的实际操作过程，是 SOP 执行评估的重要证据载体。相比文字填报和人工勾选，视频记录能够保留动作细节、操作顺序和现场状态，便于事后追溯。对于需要培训考核、过程留痕或质量监管的场景，视频评估可以提高检查依据的完整性。

在流程规范化管理中，视频评估主要发挥三方面作用。首先，视频能够保存连续操作过程，使检查人员能够回看具体执行细节。其次，视频能够提供比文字记录更丰富的动作、对象和顺序信息，有助于判断执行者是否真正按照规范操作。再次，视频可以作为复核依据，与评估结论、问题描述和整改意见共同形成完整的管理闭环。

视频评估并不意味着完全取代人工检查，而是将人工检查从“从头观看视频并自行判断”转变为“查看系统生成的结构化结论和关键证据”。这种方式能够减少重复劳动，提高反馈效率，并为后续统计分析提供结构化数据。

2.2 多模态大模型与视频理解技术

2.2.1 多模态大模型基本原理

多模态大模型是指能够同时处理文本、图像、视频等多种模态信息，并在统一任务中进行理解、推理和生成的模型。与传统文本模型不同，多模态模型可以接收视觉输入，并结合文字提示完成图像描述、视频理解、目标识别、动作判断和结构化问答等任务。

多模态大模型通常由大规模神经网络训练得到，其核心特点是能够将不同模态的数据映射到统一的语义空间中进行表示和计算。文本信息可以提供任务指令和语义描述，图像或视频信息可以提供视觉内容，模型通过跨模态关联理解不同信息之间的关系，从而完成单一模态模型难以完成的综合判断任务。

多模态大模型的优势在于具有较强的泛化能力和交互能力。用户可以通过自然语言描述任务要求，模型则结合输入的视觉内容进行分析并生成回答。相比传统视觉模型，多模态大模型不一定需要针对每个具体任务重新训练专用分类器，而是可以通过提示词适配图像问答、视频理解、视觉推理和结构化信息抽取等多种任务。

2.2.2 视频内容理解方法

视频内容理解是对连续视频数据进行分析，提取其中的时间、动作、对象和事件信息。传统视频理解方法通常依赖目标检测、动作识别、姿态估计、轨迹分析等算法，需要针对具体场景采集训练数据并设计模型。对于毕业设计项目而言，如果从零训练专用视频模型，不仅数据成本高，而且很难覆盖不同 SOP 场景。

基于多模态大模型的视频理解方法通常通过采样帧、视频片段或视频输入接口，将视频内容与文本指令共同输入模型，再由模型输出对视频内容的描述、判断或结构化结果。这种方式能够利用大模型已有的视觉语义理解能力，减少针对单一场景训练专用模型的成本。

在视频理解任务中，视频帧率、视频时长、关键帧选取和输入长度控制都会影响模型分析效果。合理的视频预处理能够减少无效信息，提高模型关注关键动作的概率；而多模态模型则可以在预处理结果基础上进一步完成语义理解和推理判断。

2.2.3 多模态模型在动作判断中的应用

SOP 评估中的动作判断不仅是识别画面中是否出现某个动作，还需要判断动作是否符合步骤意图、是否完整、是否发生在合理顺序中。例如，用户可能做出了相似动作但对象错误，也可能完成了某一步但顺序提前，还可能因为画面遮挡导致证据不足。这些情况仅靠简单图像分类很难准确处理。

多模态模型用于动作判断时，需要结合文本提示明确判断标准。提示词可以要求模型关注漏做、部分完成、顺序颠倒、过早执行、延后执行、重复操作、动作错误、证据不足和前置条件缺失等问题，也可以要求模型给出判断依据，使评估结果具备一定可解释性。

由于动作判断结果会受到拍摄角度、画面清晰度、遮挡情况和模型理解能力影响，因此在实际应用中通常需要结合规则校验、人工复核或阈值判断共同使用。多模态模型适合承担初步理解和证据提取工作，而最终结论仍需要与业务规则相结合。

2.3 前端开发工具

2.3.1 Visual Studio Code

Visual Studio Code 是常用的轻量级代码编辑器，支持 JavaScript、Vue、CSS、Python、Markdown 等多种语言和文件类型。前端开发过程中，Visual Studio Code 可以配合 Vue 语法高亮、ESLint、格式化插件和终端工具，提高页面开发和调试效率。

Visual Studio Code 具有跨平台、启动速度快、插件生态丰富等特点。开发者可以根据不同语言和框架安装对应插件，从而获得语法提示、代码格式化、错误检查、版本控制和调试等能力。其内置终端也便于执行依赖安装、项目启动和构建命令，因此在 Web 前端开发中具有较高的使用率。

2.3.2 Vue 3 前端开发框架

Vue 3 是一个渐进式 JavaScript 前端框架，适合构建单页应用。它基于标准 Web 技术，提供声明式渲染和组件化开发模型，使开发者能够将复杂页面拆分为多个结构清晰、职责明确的组件。

Vue 3 的响应式系统能够自动追踪数据变化，并将变化同步到页面视图中，从而减少手动操作 DOM 的复杂度。其组合式 API 提供了更加灵活的逻辑组织方式，有利于复用状态逻辑和拆分复杂功能。Vue 生态中还包含路由管理、状态管理、构建工具和组件库等配套方案，可以支持中小型到大型前端应用的开发。

2.3.3 Element Plus 组件库

Element Plus 是面向 Vue 3 的桌面端组件库，提供表单、按钮、表格、弹窗、上传、选择器、标签、加载状态和消息提示等常用组件。使用成熟组件库可以减少基础 UI 组件的重复开发，使开发重点集中在业务逻辑和交互流程上。

Element Plus 具有组件种类丰富、文档完善、风格统一、交互状态完整等特点。开发者可以直接使用其表单校验、弹窗反馈、表格展示和上传控件等能力，提升页面开发效率和界面一致性。对于后台管理类系统而言，Element Plus 提供的布局、数据展示和输入控件能够较好地满足常见业务界面需求。

2.4 后端开发工具

2.4.1 PyCharm

PyCharm 是面向 Python 开发的集成开发环境，支持代码补全、语法检查、断点调试、虚拟环境管理、单元测试运行和项目结构浏览。相比普通文本编辑器，集成开发环境能够将编码、运行、调试和测试集中在同一工作界面中，从而提高 Python 项目开发效率。

PyCharm 对 Python Web 开发、数据库连接、依赖管理和版本控制均提供了较好的支持。其智能提示和代码导航能力能够帮助开发者快速定位函数、类和模块之间的关系；调试工具则可以在程序运行过程中查看变量值、调用栈和异常信息，便于排查后端程序问题。

2.4.2 FastAPI 后端开发框架

FastAPI 是基于 Python 的高性能 Web 框架，适合构建 RESTful API 服务。它以类型注解为基础，能够自动完成请求参数解析、数据校验和接口文档生成。FastAPI 底层基于 Starlette 和 Pydantic，兼具较高的运行性能和良好的开发体验。

FastAPI 支持同步和异步两种接口编写方式，能够适应普通业务请求和耗时网络请求等不同场景。其依赖注入机制可以用于封装认证、权限校验、数据库连接等公共逻辑；自动生成的 OpenAPI 文档则方便开发者进行接口调试和前后端联调。因此，FastAPI 在现代 Python Web 服务开发中应用广泛。

2.4.3 Pydantic 数据校验工具

Pydantic 是 Python 中常用的数据校验和模型定义工具。FastAPI 与 Pydantic 深度集成，可以根据 Pydantic 模型自动校验请求体、生成接口文档，并将输入数据转换为标准 Python 对象。

Pydantic 通过类型声明定义数据结构，能够对字符串、数字、布尔值、列表、字典和嵌套对象等数据进行自动转换和校验。当输入数据不符合模型定义时，Pydantic 可以生成明确的错误信息，帮助开发者快速发现数据格式问题。对于接口请求、配置项、数据库对象和模型输出等结构化数据较多的应用，Pydantic 能够提高数据处理的规范性和可靠性。

2.5 视频处理技术

2.5.1 OpenCV

OpenCV 是常用的计算机视觉和图像处理库，提供图像读取、图像变换、视频读取、帧定位、特征提取和图像编码等能力。它支持 Python、C++、Java 等多种语言接口，具有较强的跨平台能力。

在视频处理领域，OpenCV 可以打开常见格式的视频文件，读取帧率、帧数和分辨率等基本信息，并按照帧序号或时间位置提取图像帧。对于需要进行关键帧提取、图像缩放、格式转换和基础视觉处理的应用，OpenCV 提供了较为完整的工具函数。无界面版本 opencv-python-headless 更适合部署在服务器端环境中。

2.5.2 FFmpeg

FFmpeg 是广泛使用的音视频处理工具，支持视频转码、封装格式转换、音频处理、视频压缩、视频裁剪和流媒体处理等功能。它支持大量音视频编码格式和容器格式，是多媒体处理领域常用的基础工具。

FFmpeg 可以通过命令行方式调用，也可以被其他语言或库间接调用。在 Web 应用中，用户上传的视频可能存在编码格式、封装格式或播放兼容性差异，使用 FFmpeg 可以将视频转换为更通用的格式，提高浏览器播放和后端处理的稳定性。与 OpenCV 相比，FFmpeg 更侧重于音视频编解码和格式转换，而 OpenCV 更侧重于帧级图像处理。

2.6 数据库技术

2.6.1 MySQL 数据库

MySQL 是成熟的开源关系型数据库管理系统，适合保存结构化业务数据。它使用 SQL 语言进行数据定义、查询、插入、更新和删除，具有较好的通用性和易用性。MySQL 支持多种存储引擎，其中 InnoDB 支持事务、外键和行级锁，能够满足多数业务系统对数据一致性和可靠性的要求。

MySQL 具有性能稳定、生态成熟、跨平台部署方便等特点，被广泛应用于网站、管理系统和企业级应用中。对于存在多张表、多种实体关系和较多查询需求的业务系统，关系型数据库可以通过主键、外键、索引和事务机制保证数据组织的清晰性和操作的可靠性。

2.6.2 PyMySQL 数据库连接工具

PyMySQL 是 Python 操作 MySQL 的常用驱动库，能够让 Python 程序连接 MySQL 数据库并执行 SQL 语句。它以纯 Python 方式实现，安装和使用较为方便，适合在 Python Web 应用中完成数据库访问。

PyMySQL 提供连接创建、游标操作、参数化查询、事务提交和回滚等基础能力。通过参数化 SQL 可以降低字符串拼接带来的安全风险；通过事务控制可以保证多条数据库操作的一致性。对于需要直接编写 SQL 并精细控制查询过程的场景，PyMySQL 是较轻量且实用的数据库连接工具。

2.7 大模型与接口

2.7.1 Qwen 3.6 Plus 多模态大模型

Qwen 3.6 Plus 是阿里云百炼提供的通义千问系列多模态模型，支持文本、图像、视频等多种输入形式，适合执行视觉理解、视频分析、复杂指令理解和结构化输出等任务。根据百炼模型文档，qwen3.6-plus 和 qwen3.5-plus 均属于支持视觉理解能力的模型，可用于处理图片、视频和文本混合输入，并能够通过模型接口返回自然语言或结构化结果。

Qwen 系列模型具有较强的中文理解能力和多模态输入处理能力，可以根据文本提示对视觉内容进行描述、分析和推理。对于包含步骤说明、视频画面和结构化输出要求的任务，多模态模型能够在同一上下文中综合处理不同类型的信息，从而生成较完整的判断结果。

从模型选型角度看，qwen3.6-plus 更侧重较强的综合理解与推理能力，qwen3.5-plus 则在价格方面更具优势，适合成本敏感或调用频率较高的场景。第二章作为技术介绍章节，只需说明模型能力和接口特点；若需要展开两种模型的价格对比、成本估算和选型理由，更适合放在第三章“经济可行性分析”中。如果后续结合实际测试数据比较两种模型的耗时、费用和评估效果，也可以在第五章测试与实验验证部分进行说明。

2.7.2 OpenAI Chat Completions 兼容接口

OpenAI Chat Completions 兼容接口是一类采用 chat/completions 请求格式的大模型调用接口。该接口通常通过 messages 数组组织对话上下文，通过 model 字段指定模型名称，通过 temperature、response_format 等参数控制生成行为，并使用 Bearer Token 方式进行身份认证。

兼容接口的优势在于调用格式统一，开发者可以使用相似的请求结构接入不同模型服务，降低模型切换和接口适配成本。对于多模态模型，接口内容中还可以包含文本、图片或视频等不同类型的数据块，使模型能够在同一次请求中处理多种输入。

在使用兼容接口时，需要注意不同模型在消息格式、思考模式、结构化输出和多模态输入方面可能存在差异。调用前应根据模型文档确认支持的参数范围，并在程序中做好超时控制、错误处理和响应解析，以保证接口调用的稳定性。

2.7.3 结构化 Prompt 与 JSON 输出约束

结构化 Prompt 是引导大模型按照指定格式输出内容的重要方法。普通自然语言回复虽然表达灵活，但不便于程序解析和后续处理；结构化 Prompt 则通过明确字段名称、字段含义、输出格式和约束条件，使模型结果更容易被系统读取和使用。

JSON 是常见的结构化数据格式，具有层次清晰、易于解析、跨语言通用等特点。在大模型接口中，可以通过提示词说明或 response_format 参数要求模型按照 JSON Schema 输出，从而限制字段类型、必填字段、数组结构和枚举范围。这样能够减少模型输出自由文本带来的解析困难。

结构化 Prompt 的设计通常需要包括角色设定、任务说明、输入信息、判断标准和输出格式等部分。对于复杂任务，还可以将问题拆分为多个阶段，分别完成信息提取、判断分析和结果汇总。合理的 Prompt 设计可以提升模型输出的稳定性，使模型更符合业务系统对格式和内容的要求。

需要注意的是，结构化输出只能提高格式稳定性，并不能完全消除模型理解误差。因此，在实际应用中仍需要结合结果校验、异常处理、规则修正和人工复核等机制，共同提高系统输出的可靠性。
